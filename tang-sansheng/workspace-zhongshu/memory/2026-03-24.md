### 三省制权限设计原理

```
皇帝（陛下）
   ↓
中书省（起草）→ 门下省（审核）→ 尚书省（派发）→ 六部（执行）
   ↓                  ↓                  ↓              ↓
 ❌ 无 spawn      ❌ 无 spawn       ✅ 有 spawn      ❌ 无 spawn
   ↓                  ↓                  ↓              ↓
只负责起草        只负责审核         负责派发         负责执行
```

### 关键认知

1. **中书省无 spawn 权限是正确设计**
   - 目的：防止中书省绕过审核直接派发
   - 体现三省制的分权制衡原则

2. **尚书省是唯一有 spawn 权限的部门**
   - 目的：统一调度，避免多头指挥
   - 六部只对尚书省负责

3. **正确的协作方式**
   ```javascript
   // 中书省应该：
   message({
     accountId: "menxia",
     message: "【诏令草案】请门下省审核..."
   })
   
   // 门下省审核后，通知尚书省：
   message({
     accountId: "shangshu",
     message: "【审核通过】请尚书省派发..."
   })
   
   // 尚书省 spawn 六部：
   sessions_spawn({
     agentId: "bingbu",
     task: "开发后端..."
   ```

2. ❌ 没有理解三省制的分权制衡原理

## Message工具单向通知特性（2026-03-24 09:06 更新）

### 工具机制深度理解

**Message工具是单向通知，不会触发Agent自动回复！**

### 工作原理

```
中书省 → message → 其他部门
  ↓
✅ 消息已送达（有messageId为证）
❌ 但不会触发自动回复
```

### 类比理解

```
Message工具 = 发送邮件（单向）
对话/spawn = 打电话（双向）
```

### 实际验证

**通知送达情况**：
- 门下省：3次通知，messageId全部正常
- 尚书省：3次通知，messageId全部正常
- 其他7个部门：全部送达

**messageId示例**：
```
门下省 - 文件存放规范：nh1Mqwj2vUKOqJ6h9V6tDnhwShZGfi4EWF4RJ7HaXZU=
门下省 - CLAUDE.md更新：wQVG+/yIZX6nDEPSJGUOZK1/U94MHQX2YNqh7R8fkXc=
门下省 - 项目位置更新：pkXsirnTsyqFt7/vNjBjCGVgNVPLwB62OueYLg8UXzU=
```

### 获得回复的正确方式

#### 方式1：陛下主动对话
```
1. 打开钉钉
2. 找到对应部门机器人
3. 发送消息
4. 部门会回复
```

#### 方式2：通过尚书省spawn
```javascript
sessions_spawn({
  agentId: "部门ID",
  task: "任务内容"
})
```

#### 方式3：等待下次对话
```
下次陛下与部门对话时：
- Agent启动 → 读取AGENTS.md
- 看到规范 → 自动遵守
```

### 适用场景

**Message工具**：
- ✅ 单向通知
- ✅ 广播消息
- ✅ 规范传达
- ❌ 不适合获取回复

**对话/spawn**：
- ✅ 获取回复
- ✅ 任务执行
- ✅ 交互式对话

### 经验总结

1. **Message不会触发回复** - 这是设计如此，不是问题
2. **messageId确认送达** - 有messageId就是成功
3. **需要回复要主动对话** - 陛下主动找部门
4. **规范持久化到AGENTS.md** - 确保下次对话自动遵守
