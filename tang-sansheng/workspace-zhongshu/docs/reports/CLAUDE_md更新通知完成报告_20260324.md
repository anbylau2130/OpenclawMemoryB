## 📋 CLAUDE.md 更新内容

### 核心规范

#### 1. 语言要求
```
✅ 使用中文回答问题
✅ 所有文档和注释都使用中文
```

#### 2. 开发原则
```
✅ 实现功能之前考虑是否有适合的类库
✅ 简化响应，分步处理，避免长代码
✅ 每次生成代码后重构代码，尽量使用设计模式
✅ 进行模块化开发
```

#### 3. 代码规范
```
✅ 保证每个类的功能都是单一的
✅ 每个方法长度不超过50行
✅ 不破坏现有项目的结构
✅ 每次生成成功后校验编译错误
```

#### 4. 编码规范
```
✅ 严格遵守项目语言的编码规范和命名约定
✅ 使用有意义的变量和方法名称，遵循驼峰命名法
✅ 合理使用注释，保持代码可读性
✅ 正确处理异常，使用try-catch包装可能出错的代码
```

#### 5. 架构规范
```
✅ 避免在循环中进行数据库操作
✅ 使用依赖注入而不是硬编码服务
✅ 避免在控制器中直接使用数据库上下文
✅ 遵循分层架构，控制器负责协调，业务逻辑在服务层
```

#### 6. 全局思维
```
✅ 修改代码的时候从全局角度去改代码
✅ 不要只改一个文件，导致其他文件异常
✅ 类，结构，枚举等类型的定义需要写入单独的文件
✅ 每个文件都是功能单一的，不存在一个文件中包含过个类型
```

#### 7. 质量保证
```
✅ 保证所有项目都能正常编译
✅ 检查运行时异常
✅ 保证所有代码都能正常工作
✅ 避免可预测的运行时异常
```

#### 8. 探索优先
```
✅ 功能修改之前必须探索项目代码
✅ 从全局考虑修改
✅ 避免局部修改，全局出问题
```

#### 9. Blazor 组件规范
```
✅ 标签代码写在 [组件名].razor
✅ C#代码写在 [组件名].razor.cs
✅ CSS样式代码写在 [组件名].razor.css
✅ 保持代码语言代码的分离
✅ 不要让组件代码行数太多
✅ 保持每个组件功能的单一性和可复用性
```
