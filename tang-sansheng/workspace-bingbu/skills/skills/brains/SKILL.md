# Brains - 头脑风暴与创意思维

## 概述

在开始任何创造性工作之前，通过协作式对话将想法转化为完整的设计和规格。

## 核心流程

### 1. 理解想法
- 先检查当前项目状态（文件、文档、最近提交）
- 一次问一个问题来细化想法
- 尽可能使用多选题，但开放式问题也可以
- 每条消息只问一个问题
- 重点关注：目的、约束、成功标准

### 2. 探索方案
- 提出 2-3 种不同的方案及其权衡
- 用对话方式呈现选项，给出推荐和理由
- 首先提出推荐选项并解释原因

### 3. 呈现设计
- 一旦理解了要构建什么，就呈现设计
- 分段呈现（每段 200-300 字）
- 每段后确认是否正确
- 如果某段有问题，暂停并细化

## 使用场景

### 功能设计
```
用户：我想添加一个用户登录功能
Brains：
1. 先问：需要支持哪些登录方式？
   - 邮箱密码
   - 手机验证码
   - 第三方登录（微信、GitHub）
   - 其他

2. 探索方案：
   - 方案A：JWT Token（推荐）
   - 方案B：Session + Cookie
   - 方案C：OAuth 2.0

3. 呈现设计（分段）
```

### 架构设计
```
用户：我想设计一个微服务架构
Brains：
1. 先问：系统规模和团队情况？
2. 探索：单体 vs 微服务 vs Serverless
3. 呈现：架构图 + 技术栈选择
```

### 问题解决
```
用户：系统性能太慢了
Brains：
1. 先问：瓶颈在哪里？（数据库、API、前端）
2. 探索：优化方案（缓存、索引、CDN）
3. 呈现：优化计划
```

## 提问技巧

### 好的提问
- ✅ "用户量级是多少？（100、1000、10000+）"
- ✅ "性能要求是什么？（<100ms、<1s、无要求）"
- ✅ "预算有限制吗？"

### 避免的提问
- ❌ "你想要什么？"（太开放）
- ❌ "技术栈是什么？"（应该先检查项目）
- ❌ 一次问多个问题

## 设计呈现格式

### 1. 概述（200-300 字）
```markdown
## 设计概述

### 目标
- 主要目标
- 次要目标

### 约束
- 技术约束
- 业务约束
- 时间约束
```

### 2. 架构设计（200-300 字）
```markdown
## 架构设计

### 系统架构
[架构图或描述]

### 核心组件
- 组件1：职责
- 组件2：职责
```

### 3. 技术选型（200-300 字）
```markdown
## 技术选型

### 前端
- 框架：React/Vue/Svelte
- 理由：...

### 后端
- 语言：Node.js/Python/Go
- 框架：Express/FastAPI/Gin
- 理由：...
```

### 4. 实现计划（200-300 字）
```markdown
## 实现计划

### Phase 1: 基础功能（1-2周）
- [ ] 任务1
- [ ] 任务2

### Phase 2: 高级功能（2-3周）
- [ ] 任务3
```

## 最佳实践

### 1. 检查上下文
```bash
# 查看项目文件
ls -la

# 查看最近提交
git log --oneline -10

# 查看文档
cat README.md
```

### 2. 一次一个问题
```
❌ 错误：
"你想用什么技术栈？数据库用什么？部署到哪里？"

✅ 正确：
"你想用什么前端框架？（React/Vue/Svelte/其他）"
```

### 3. 分段呈现
```
❌ 错误：
一次性呈现 10 页设计文档

✅ 正确：
分段呈现，每段后确认：
"这部分看起来对吗？"
```

## 示例对话

### 示例 1：功能设计

**用户：** 我想添加一个搜索功能

**Brains：** 
好的！让我先了解一下当前项目。

[检查项目文件]

几个问题帮助我理解需求：

1. **搜索范围**是什么？
   - 全局搜索（所有内容）
   - 局部搜索（特定模块）
   - 文件搜索
   - 其他

**用户：** 全局搜索

**Brains：**
明白了，全局搜索。

2. **数据量级**大约是多少？
   - < 1000 条记录
   - 1000 - 10000 条
   - 10000 - 100000 条
   - > 100000 条

**用户：** 10000 - 100000 条

**Brains：**
好的，中等规模数据。

基于你的需求，我推荐 3 种方案：

**方案 A：Elasticsearch（推荐）**
- ✅ 性能优秀（< 100ms）
- ✅ 功能强大（全文搜索、聚合）
- ❌ 需要额外部署
- 适合：数据量大、搜索复杂

**方案 B：数据库全文索引**
- ✅ 无需额外组件
- ✅ 实现简单
- ❌ 性能一般（> 1s）
- 适合：数据量中等、搜索简单

**方案 C：MeiliSearch**
- ✅ 轻量级
- ✅ 性能好（< 200ms）
- ❌ 功能有限
- 适合：中小型项目

**我推荐方案 A（Elasticsearch）**，因为你的数据量在 1-10 万条，需要良好的搜索性能。

这个方案适合你吗？还是想了解其他方案？

## 常见问题

### Q: 什么时候使用 Brains？
A: 在开始任何创造性工作之前：
- 创建新功能
- 构建组件
- 添加功能
- 修改行为

### Q: 如果用户已经有明确想法怎么办？
A: 仍然使用 Brains 来验证和完善想法：
- 确认理解正确
- 探索替代方案
- 细化实现细节

### Q: 分段呈现太慢怎么办？
A: 可以根据用户反馈调整：
- 如果用户着急，可以加快节奏
- 如果设计复杂，必须分段确认
